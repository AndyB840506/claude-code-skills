# Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Connection refused on :8188 | Server not running | Launch per `stack-reference.md`; wait with curl retry pattern |
| "Missing Models (N)" panel in UI | Workflow needs files not present | Download each, place per `add-model.md`, press `R`, re-select in dropdowns (may show "undefined") |
| New model not in dropdown | Not refreshed / wrong folder / yaml mapping added after startup | `R` first → check folder vs table → restart server |
| CUDA out of memory | 12 GB card + big model + other GPU apps | Close games/video apps; lower resolution; switch to fp8 variant of the model |
| Every generation slow (not just first) | Model spilling to system RAM | Same as OOM: smaller/fp8 model variant |
| First generation slow (1–3 min) | Checkpoint loading — NORMAL | No action; don't diagnose before one full run completes |
| c10.dll error at startup | Missing VC redist | https://aka.ms/vc14/vc_redist.x64.exe |
| `tar: This does not look like a tar archive` on .7z | Git Bash GNU tar | Use `& "$env:WINDIR\System32\tar.exe" -xf` (bsdtar) |
| Downloaded file "missing" | It's in `E:\Downloads`, not the user profile Downloads | Check `E:\Downloads` first |
| Bad/garbled generations from anime checkpoint | Natural-language prompt on booru-tag model | Rewrite as comma-separated tags (`1girl, ...`) |
| JSON parse errors from API | BOM or empty response | Global rule: check BOM + empty body specifically |
| Quality below the model's reputation after replicating an official recipe | A component was substituted (e.g. standard t5xxl instead of the flan variant the creator ships) | Match EVERY file of the official example (encoders included) or flag the substitution as a risk up front — bit 2026-07-11 (Chroma) |
| "Model looks bad" dispute — is it the recipe, the quant, or the prompt? | Multiple variables changed at once | Same-seed A/B via API: one payload per variable (control vs candidate), compare outputs side by side. 2026-07-11: verdict in ~15 min — prompt density dominated, recipe changes were marginal |
| Chroma1-HD output is pure static/noise (status still reports "success") | Used `ModelSamplingSD3` instead of `ModelSamplingAuraFlow` to wrap the UNET, and/or decoded `SamplerCustomAdvanced` output slot 1 (`denoised_output`) instead of slot 0 (`output`) | Match the user's verified workflow `repos\comfyui-setup\workflows\Chroma Personajes Pro.json` node-for-node: `ModelSamplingAuraFlow` (shift 1.0), a `T5TokenizerOptions` node (min_padding=1) between `CLIPLoader` and both `CLIPTextEncode`, and `VAEDecode` reading `SamplerCustomAdvanced` slot 0. `node_errors` being empty and `status_str: "success"` only mean the graph ran without exceptions — always Read the resulting PNG before trusting it (bit 2026-07-15, building `comfyui/templates/chroma-txt2img-api.json`) |
| Generated images not found in `E:\AI\outputs\` after a run that reported success | Server was launched via `run_nvidia_gpu.bat` (the user's quick-launch script, used by `imagen-a-prompt/workflows/captioner-local.md`), which does NOT pass `--output-directory` | Check `E:\AI\ComfyUI_windows_portable\ComfyUI\output\` instead — that launcher's images land in ComfyUI's internal default output folder. Only the headless launch command in `docs/stack-reference.md` (with `--output-directory E:\AI\outputs`) writes to the documented path. Confirm which launcher is running before searching for output files (bit 2026-07-15, validating the 4 new templates) |
| `POST /prompt` returns generic `500 Internal Server Error` / "Server got itself in trouble" with no useful detail | The HTTP error body is generic by design | Check `E:\AI\comfyui-stderr.log` (or the console the server was launched from) — it has the real Python traceback. Common cause: malformed API-format JSON authored by hand (e.g. a missing closing brace) — validate with `json.load()` before POSTing (bit 2026-07-21, BTQ EP.023) |
| Windows path argument to a launched process comes out mangled (e.g. `--output-directory E:\AI\outputs` arrives as `E:AIoutputs`, backslashes gone) | Git Bash silently eats single backslashes in unquoted arguments that aren't a recognized escape sequence | Launch Windows executables with backslash paths via the PowerShell tool (`Start-Process`), not Bash — confirm the fix by checking `argv` in `/system_stats` (bit 2026-07-21, BTQ EP.023) |
| Banned motif (e.g. a vetoed concentric-ring/target pattern) reappears even after an explicit "DO NOT render X" in the prompt | Not a Flow-only quirk — confirmed in local Z-Image Turbo generation too (BTQ EP.022, 2026-07-20: a clock scene got concentric rings in the background, a battery scene got a bullseye logo baked onto it, on the first AND second attempt) | Reinforce the prompt with the specific object/material explicitly forbidden (e.g. "no printed graphics, no logo, no circles of any kind on its surface — completely plain") rather than repeating the same generic ban; may take 2-3 rounds. **It can also hide as texture, not just a bold graphic**: a fabric/material surface can render as dozens of tiny repeated rings — invisible when eyeballing the full downscaled image, only visible after cropping/zooming into that region with PIL before approving. Always zoom into busy textures/materials near the frame edges, not just the overall composition, before signing off |
